# 02 — 如何设计 Agent 结构

> **这不是固定配置。** 以下方案是设计思路的示例，不是要求你照抄。
> 你应该根据自己的模型、skill、预算和任务类型选择或组合。

---

## 方案 A：极简 2-agent

**适合**：skill 较少、任务类型单一、刚开始使用 Multica 的用户。

| Agent | 职责 | 推荐模型 |
|-------|------|---------|
| **Manager** | 理解任务、路由判断、创建 issue、汇总结果 | 主力模型 |
| **Worker** | 执行所有类型的任务 | 主力模型 |

**优点**：简单、不需要分诊逻辑。
**缺点**：所有任务走同一个 worker，无法按成本/能力分层。

---

## 方案 B：3-agent 能力分层

**适合**：有多模型、想按能力分配任务的用户。

| Agent | 职责 | 推荐模型 |
|-------|------|---------|
| **Light** | 轻任务、分诊、初稿、整理 | 经济模型（Flash） |
| **Main** | 主力执行 | 主力模型（Pro） |
| **Review** | 高质量审查 | 最强模型（GPT/Opus） |

**优点**：按能力分层，经济和效率兼顾。
**缺点**：需要明确每个任务的复杂度判断标准。

---

## 方案 C：按领域拆分

**适合**：任务类型多样的用户。

| Agent | 职责 |
|-------|------|
| **Writing Agent** | 写作、润色、翻译、整理 |
| **Coding Agent** | 编程、调试、审查、文档 |
| **Research Agent** | 调研、分析、综述 |
| **Review Agent** | 安全和质量审查 |

**优点**：领域清晰，各司其职。
**缺点**：agent 较多，可能需要 manager。

---

## 方案 D：写作 + 编程双线 × 三层

**适合**：任务种类多、模型多、希望精细分工的用户。

### 写作线

| Agent | 推荐模型 |
|-------|---------|
| Writing-Light | 经济模型 |
| Writing-Main | 主力模型 |
| Writing-Review | 最强模型 |

### 编程线

| Agent | 推荐模型 |
|-------|---------|
| Coding-Light | 经济模型 |
| Coding-Main | 主力模型 |
| Coding-Review | 最强模型 |

**优点**：精细分工，成本最优。
**缺点**：agent 数量多，强烈建议搭配 manager。

---

## 方案 E：作者个人的案例（Example Case Study）

作者的配置是方案 D 的变体，使用 1 个 manager + 3 个 coding + 3 个 writing = 7 个 agent。

详见 `docs/07-case-study-my-setup.zh.md`。

**再次强调：这只是一个案例，不是必须照抄。** 作者的模型、skill 和任务类型决定了这个结构。你的情况可能完全不同。

---

## 选择建议

| 你的情况 | 建议方案 |
|---------|---------|
| 刚开始，skill < 5 | A（2-agent）或 B（3-agent） |
| 写作 + 编程都做，模型有两个以上 | B（3-agent）或 C（按领域拆分） |
| 任务精细，追求成本最优 | D（双线三层） |
| 不确定 | 从 A 开始，3-5 个任务后再调整 |
