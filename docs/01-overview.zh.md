# 01 — 问题总览

## 这个项目解决什么问题？

Multica 是一个强大的 managed agents 平台，但随着 skill 和 agent 增多，会出现以下问题：

### 问题 1：Skill 太多，不知道什么时候用

安装了 10+ 个 skill 后，面对一个具体任务，不知道该加载哪个。全部加载会增加 token 消耗和判断混乱，只加载一部分又怕漏掉有用的。

### 问题 2：英文 skill 名称和描述难懂

很多 skill 的 description 只有英文，对中文用户不够友好。中文用户需要一个清晰的中文索引来快速判断。

### 问题 3：多模型能力和成本不同，任务分配混乱

如果你有多个模型（例如便宜的 Flash、主力的 Pro、最强的 GPT 或 Claude Opus），它们的价格、能力和额度差异很大。不是所有任务都值得用最强模型。

### 问题 4：强模型有额度限制，不适合做所有任务

GPT-5.5、Claude Opus 等最强模型通常有限额。把它们用在简单格式化和短文润色上是浪费。

### 问题 5：需要一个方法把 agent、skill、issue 派单串起来

Multica 的核心协作方式是 issue-based，不是同步函数调用。需要一套方法让 agent 之间稳定协作，而不是假设能"直接调用"另一个 agent。

---

## 本项目的解决思路

1. **先理解自己的资源**：模型、skill、预算
2. **再设计 agent 结构**：数量、分工、命名
3. **配置路由层**：skill router + CLI operator
4. **建立测试流程**：从低风险开始验证
5. **迭代优化**：根据实际使用调整

---

## 适用和不适用

**适用：**
- 你有 3 个以上 skill 需要管理
- 你有多个模型，能力和成本不同
- 你想让 agent 协作而非各自为政
- 你想把自己的工作流整理成可复用的模板

**不适用：**
- 你只有 1-2 个 skill
- 你只有一个模型
- 你不需要 agent 协作
- 你只需要一个 agent 完成所有任务
